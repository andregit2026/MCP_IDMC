MCP vs REST API

Understanding the Difference: Two Ways to Access IDMC

What We're Actually Doing

Quick Answer: In our comparison script, we used direct REST API calls, not MCP calls. But the MCP server we built wraps those same REST APIs so AI assistants (like Claude) can use them conversationally.

🌐 REST API (Direct)

What it is: Standard HTTP web API provided by Informatica IDMC

  • Direct HTTP requests to IDMC servers
  • Uses libraries like Python's requests
  • You manually handle authentication & parsing
  • Standard for all programming languages

🤖 MCP (Wrapper)

What it is: Model Context Protocol - a standardized interface for AI tools

  • Wraps REST APIs for AI assistants
  • Claude can call functions directly
  • Auto-handles authentication & errors
  • Designed specifically for AI interaction

Key Concepts Explained

🌐 REST API (What Informatica Provides)

REST = Representational State Transfer

  • Standard web API architecture used across the internet
  • Uses HTTP methods: GET, POST, PUT, DELETE
  • Returns data in JSON format
  • Requires authentication (session IDs, tokens)
  • Everyone can use - Python, JavaScript, curl, Java, etc.

Example with curl:

curl -X GET \ 'https://emc1.dm1-em.informaticacloud.com/saas/api/v2/mapping/010MEP1700000000000Z' \ -H 'icSessionId: your-session-id'

🤖 MCP (What We Built as a Wrapper)

MCP = Model Context Protocol (by Anthropic)

  • A standardized way to give AI assistants tools to use
  • Wraps REST APIs in a simple function interface
  • Makes it easy for AI to interact with external systems
  • Handles authentication and error formatting automatically
  • Designed specifically for AI assistants, not general programming

When to Use Each Approach

Use MCP Server When:

  • You want Claude to interact with IDMC
  • Building AI-powered workflows
  • Need conversational access to data
  • Want automatic auth and error handling
  • Exploring data interactively

Use Direct REST API When:

  • Building standalone scripts
  • Need fine-grained control
  • Integrating with other systems
  • Automation & batch processing
  • CI/CD pipelines
  • Performance-critical operations

Summary

In Our Comparison Script:

❌ We did NOT use MCP calls

✅ We made direct REST API calls using Python's requests library

This was appropriate for a standalone analysis script that needed detailed control

The MCP Server (server.py):

✅ Provides MCP tools that wrap those same REST APIs

✅ Allows Claude to interact with IDMC conversationally

Example: Instead of writing code, you can ask Claude "list all mappings" and Claude uses the MCP tool

Key Insight:

Both access the same IDMC REST APIs underneath, just through different interfaces!

  • REST API: You write code to make HTTP requests
  • MCP: Claude calls functions that make those HTTP requests for you

Real-World Examples

Scenario 1: Automated Daily Report

Use REST API

You need a Python script that runs every night via cron, fetches all mapping execution logs, analyzes failures, and emails a report. Direct REST API gives you full control and reliability for automation.

Scenario 2: Interactive Data Exploration

Use MCP Server

You're chatting with Claude: "Show me all mappings that failed yesterday." Claude uses the MCP tools to query IDMC and presents results conversationally. No coding needed!

Scenario 3: CI/CD Pipeline Integration

Use REST API

Your Jenkins pipeline needs to deploy a mapping, run it, and verify success before merging code. Direct REST API calls give you precise control over the deployment workflow.

Scenario 4: Ad-hoc Investigation

Use MCP Server

Something failed in production. You ask Claude: "What mappings depend on connection X?" Claude uses MCP tools to quickly gather information without writing scripts.
